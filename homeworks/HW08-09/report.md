# HW08-09 – PyTorch MLP: регуляризация и оптимизация обучения

## 1. Кратко: что сделано

- Выбран датасет **KMNIST (вариант A)**.
- В части A выполнены эксперименты E1-E4, в части B — O1-O3.

## 2. Среда и воспроизводимость

- Python: 3.12.10
- torch / torchvision: 2.10.0+cpu / 0.25.0+cpu
- Устройство (CPU/GPU): cpu
- Seed: 42
- Как запустить: открыть `HW08-09.ipynb` и выполнить Run All.

## 3. Данные

- Датасет: KMNIST
- Разделение: train/val/test (train split 80/20 + test из torchvision)
- Трансформации (transform): `ToTensor()` + `Normalize((0.5,), (0.5,))`
- Комментарий: это 10-классовый датасет 28x28; для MLP хорошо виден эффект регуляризации и выбора lr.

## 4. Базовая модель и обучение

- Модель MLP (кратко): 2 скрытых слоя `[256, 128]`, ReLU
- Loss: CrossEntropyLoss
- Базовый Optimizer (для части A): Adam (`lr=0.001`)
- Batch size: 240
- Epochs (макс): 4 для E1-E3, 7 для E4
- EarlyStopping: patience=2, отбор по val_accuracy

## 5. Часть A (S08): регуляризация (E1-E4)

- E1 (base): 2 скрытых слоя, без Dropout/BatchNorm
- E2 (Dropout): как E1 + Dropout(p=0.25)
- E3 (BatchNorm): как E1 + BatchNorm
- E4 (EarlyStopping): лучший из (E2/E3) + EarlyStopping

## 6. Часть B (S09): LR, оптимизаторы, weight decay (O1-O3)

- O1: LR слишком большой (Adam, `lr=5e-2`)
- O2: LR слишком маленький (Adam, `lr=5e-6`)
- O3: SGD+momentum (`momentum=0.9`) + `weight_decay=2e-4` (`lr=8e-3`)

Короткая сводка (5-9 строк):

- Лучший эксперимент части A: E4
- Лучшая `val_accuracy`: 0.8208
- Итоговая `test_accuracy` (для лучшей модели): 0.6279
- На O1 (слишком большой LR) обучение идёт быстрее, но динамика менее стабильная.
- На O2 (слишком маленький LR) обучение почти не сдвигается.
- O3 (SGD+momentum+weight_decay) в коротком прогоне уступает лучшему Adam-конфигу без доп. тюнинга.

## 8. Анализ

В этой версии BatchNorm дал более устойчивый результат, чем Dropout. EarlyStopping помог остановить обучение в момент, когда метрика перестала улучшаться. Для O2 с очень маленьким шагом скорость оптимизации недостаточна, и это видно по слабой динамике loss/accuracy. Для O1 с большим шагом обучение быстрое, но более шумное. Конфиг O3 может быть полезен для обобщения, но ему обычно нужно больше эпох и подбор lr.

## 9. Итоговый вывод

В качестве базового варианта для этой версии я бы оставил E4: `[256, 128] + BatchNorm + Adam + EarlyStopping`. Дальше можно улучшать качество за счёт обучения на большем объёме train-данных и более аккуратного подбора гиперпараметров.
