# HW14 – эмбеддинги, FAISS, оценка retrieval и mini-RAG по базе знаний

## 1. Кратко: что сделано

- В качестве базы знаний использован небольшой локальный корпус из 12 тематических документов про табличное машинное обучение, валидацию, leakage, метрики и мониторинг модели.
- Документы были разбиты на чанки скользящим окном по словам.
- В качестве векторных представлений использованы TF-IDF векторы по uni- и bi-gram, а для поиска собран индекс `FAISS`.
- Базовая проверка retrieval показала, что система стабильно находит релевантный источник в `top_k`, но на части запросов рядом появляются тематически близкие, но не самые точные документы.
- После обновления базы знаний тремя новыми документами retrieval заметно улучшился на запросах про threshold tuning, data drift и target leakage checklist.
- Mini-RAG собран в extractive-варианте: вопрос -> retrieval -> сборка контекста -> выбор самых релевантных предложений -> ответ с источниками.

## 3. База знаний и подготовка документов

- Предметная область: табличное машинное обучение, подготовка данных, метрики, leakage, calibration и мониторинг модели.
- Источник документов: локально собранный набор коротких учебных текстов внутри `HW14.ipynb`.
- Число исходных документов: `12`
- Число чанков после разбиения: `24`
- Какой способ чанкинга использовался: скользящее окно по словам.
- Основные параметры чанкинга: `chunk_size=70`, `overlap=12`
- Комментарий (3-6 предложений): выбранная база знаний достаточно узкая, чтобы retrieval можно было проверять воспроизводимо, но при этом не тривиальная, потому что темы документов пересекаются. Внутри корпуса есть близкие по смыслу материалы про preprocessing, leakage, cross-validation, metrics и calibration, поэтому retriever должен различать соседние понятия, а не только ловить одно редкое слово. Такой корпус подходит для учебного mini-RAG, потому что по нему легко задавать короткие проверяемые вопросы и отдельно анализировать промахи retrieval. При этом объем корпуса остается небольшим, поэтому ноутбук быстро выполняется целиком и не требует внешних сервисов.

## 4. Retrieval и индекс

### 4.1. Эмбеддинги и индекс

- Какая embedding-модель или способ векторизации использовались: `TfidfVectorizer(ngram_range=(1, 2))` с последующей L2-нормализацией векторов.
- Какая метрика сходства использовалась: cosine similarity через inner product по нормализованным векторам.
- Какой индекс `FAISS` использовался: `IndexFlatIP`
- Какой `top_k` использовался по умолчанию: `3`
- Что показали первые примеры retrieval: запросы про train/validation/test split, probability calibration, pipelines и error analysis корректно находили нужные документы в верхних позициях. Одновременно уже на первых примерах было видно, что рядом с правильным источником иногда появляются соседние по теме документы, например про leakage, calibration или related preprocessing steps.

### 4.2. Контрольные запросы и оценка retrieval

- Сколько контрольных запросов использовалось: `10`
- Что считалось релевантным ответом: попадание ожидаемого документа в список retrieved sources на уровне `doc_id`
- Какие метрики retrieval считались: `hit@k`, `recall@k`, `MRR@k`
- Итоговый `hit@k`: `1.00`
- Итоговый `recall@k`: `0.90`
- Дополнительные метрики (если были): `MRR@k = 0.95`
- Какие запросы оказались самыми простыми: запросы про dataset split, cross-validation, regularization и false positive / false negative
- Какие запросы оказались самыми проблемными: запрос `which metrics matter when accuracy is misleading` нашел релевантный источник только на второй позиции и не вернул второй ожидаемый документ в `top_k`; запрос `how to handle missing values without leakage` вернул только один из двух ожидаемых источников

## 7. Анализ

Выбранная база знаний оказалась удачной для учебной работы, потому что она компактная, тематически связная и при этом содержит документы с частично пересекающейся лексикой. Благодаря этому retrieval нельзя считать совсем тривиальным: система должна отличать leakage от calibration, preprocessing от validation и общие метрики от анализа ошибок. Выбранный чанкинг со скользящим окном оказался достаточным для воспроизводимого baseline. Эксперимент с `chunk_size` показал, что более крупные чанки в этой задаче оказались удобнее: `hit@k` и `recall@k` не изменились, но `MRR@k` стал лучше, а число чанков уменьшилось. Это означает, что маленькие чанки не давали заметного выигрыша в полноте, но немного ухудшали ранжирование. Формальная оценка retrieval оказалась полезной, потому что по одним только удачным примерам поиска было бы трудно заметить запросы, где нужный источник найден, но расположен слишком низко или найден не полностью. Особенно показательным оказался запрос про misleading accuracy: retrieval нашел релевантный документ, но первым выдал соседний по теме источник. Обновление базы знаний хорошо продемонстрировало идею, что retrieval нельзя улучшить только настройкой индекса, если в корпусе просто нет нужного знания. После добавления трех новых документов запросы про threshold tuning, data drift и target leakage сразу стали работать корректнее. Mini-RAG в таком виде ведет себя предсказуемо: если retrieval возвращает хороший контекст, extractive answer builder дает короткий grounded answer, а если в top-k появляется тематический шум, ответ становится частично расплывчатым. Самыми показательными ошибками оказались случаи, где близкие по теме документы конкурировали между собой. Ограничение текущего решения в том, что оно использует TF-IDF baseline и очень простой extractive генератор без полноценной генеративной модели, reranking и более строгой sentence-level post-processing. Кроме того, релевантность проверялась на уровне `doc_id`, а сам корпус небольшой и специально собранный для учебной задачи.

## 8. Итоговый вывод

В качестве базового retrieval-конвейера для такой задачи я бы оставил схему `chunking -> TF-IDF vectors -> L2 normalization -> FAISS IndexFlatIP -> top-k retrieval -> extractive mini-RAG`. Главный вывод по качеству retrieval состоит в том, что даже на маленьком учебном корпусе нужно считать метрики, а не полагаться только на визуально удачные примеры. Эксперимент с параметрами показал, что более крупный chunk в этой работе дал лучшее ранжирование без роста сложности корпуса. Самое важное наблюдение по mini-RAG состоит в том, что качество итогового ответа почти полностью определяется качеством найденного контекста. Если retrieval возвращает точные чанки, даже простой answer builder работает приемлемо; если retrieval шумит, это сразу ухудшает grounded answer.
